# Electron 桌面 App → 移动端 PWA 迁移设计

> 日期：2026-06-18
> 状态：已与用户确认（brainstorming 完成），待写实现计划
> 关联文档：[ARCHITECTURE.md](../../ARCHITECTURE.md)、[data-foundation-schema-v0.1.md](../../data-foundation-schema-v0.1.md)、[behavior-event-design.md](../../behavior-event-design.md)

---

## 1. 背景与动机

当前产品是 macOS Electron 桌面 App（Electron 41 + React 19 + TS 5 + better-sqlite3），无法在手机上使用。用户希望：

1. 调整为符合手机端操作习惯的 App，支持 iOS / Android。
2. 微信小程序或类似 Web 端应用也可接受。
3. 用户无移动端开发经验，需要科普 + 迁移建议。

经 brainstorming 确认的关键约束：

- **平台**：暂未决定，先看完整对比后选定（最终选定 PWA）。
- **数据架构**：坚持"全本地、零运维"理念（与 AGENTS.md §2.2 / §3 硬约束一致），不起后端服务器。
- **数据规模**：种子库 735MB（5148 只股票、597 万根 K 线），手机放不下，接受精简，但**不能过于激进**——希望保留更多股票数据，避免训练重复。
- **交互**：支持横竖屏双布局，参考股票 App（富途/同花顺）优化。

---

## 2. 方案选型：为什么是 PWA

对比 5 条移动端路径后的结论：

| 方案 | 代码复用 | 上架难度 | 结论 |
| --- | --- | --- | --- |
| 微信小程序 | ~50%（UI 框架换 Taro/uni-app） | 微信审核，金融类严 | 否决 |
| PWA | **~90%** | 零审核，HTTPS 即可 | **✅ 选定** |
| React Native | ~60%（UI 重写） | 双端 App Store 审核，金融类严 | 备选（未来升级路径） |
| Flutter / 原生 | <30% | 审核最严，沉没成本大 | 否决 |

**选定 PWA 的核心理由**：

1. 代码复用率最高（~90%），现有 React 组件、zustand store、tradingEngine 几乎原封不动。
2. 零审核、零运维，完全符合"全本地"理念。
3. 沉没成本最低：未来若需正式 App，从 PWA 升级到 React Native + Expo 是平滑路径，业务逻辑层不需重写。
4. K 线图表库 klinecharts 直接可用。

**已知限制（可接受）**：

- iOS Safari 对 IndexedDB 有配额限制（~设备空间 6-10%，64GB 设备约 4-6GB，本方案最大累积 654MB，远低于配额）。
- 无推送通知、不稳定后台——但训练工具"打开练一局就关"的场景不受影响。

## 3. 现状关键事实（来自代码探索）

- **代码规模**：~11,142 行 TS/TSX，40 个文件。
- **Electron API 调用**：128 处，集中在 15 个文件（`Grep` 统计）。
  - `BlindTrainingWorkbench.tsx`：32 处
  - `DataManagement.tsx`：14 处
  - `TrainingOverview.tsx`：3 处
  - `blind-workbench/ProfileManager.tsx`：6 处
  - `main/ipc/blind.ts` + `main/ipc/data.ts`：48 处 IPC
  - `stores/platformStore.ts`：8 处
- **关键利好**：前端组件已通过 `window.electronAPI` 抽象层调用（`src/preload/index.ts:114` `contextBridge.exposeInMainWorld('electronAPI', electronAPI)`），迁移时只需替换该层实现，上层组件签名不变。
- **数据库**：`better-sqlite3`（Node 原生 C++ 模块），主进程 `db.ts` / `blindDb.ts` / `marketDb.ts` 三处使用，仅用 `kline_daily` / `stock_list` 两张表。
- **种子库**：`data/blind-seed.db` 735MB，5148 只股票，597 万根 K 线（`sqlite3 COUNT` 实测）。
- **`list_date` 字段为空**：5148 只股票全部显示"2015 前"，无法按上市日期识别新股，需用 K 线最早日期替代。

## 4. 代码资产迁移分类

| 类别 | 占比 | 文件 | 处理方式 |
| --- | --- | --- | --- |
| ✅ 直接保留 | ~70% | `tradingEngine.ts`、`sampleFactory.ts`、`types.ts`、`formatters.ts`、`constants.ts`、`useCountUp.ts`、`Icons.tsx`、zustand store、IPC 类型定义 | 纯 TS/React，不碰 Electron，原封不动 |
| 🔧 改造适配 | ~20% | `BlindTrainingWorkbench.tsx`、`DataManagement.tsx`、`TrainingOverview.tsx`、`ProfileManager.tsx`、`platformStore.ts` | `window.electronAPI.xxx` → `window.mobileAPI.xxx`，签名不变 |
| 🔄 完全替换 | ~10% | `main/index.ts`、`preload/index.ts`、`main/ipc/*.ts`、`db.ts`、`blindDb.ts`、`marketDb.ts` | Electron/Node 专属，浏览器跑不了，重写为 Web 版 |

**抽象层是省力核心**：组件调用 `window.electronAPI.db.saveSession()`，迁移后改为 `window.mobileAPI.db.saveSession()`——函数名和参数完全一样，全局替换即可。

## 5. 数据库迁移：better-sqlite3 → sql.js

| 维度 | 现在（better-sqlite3） | 迁移后（sql.js） |
| --- | --- | --- |
| 运行环境 | Node 主进程（C++ 原生模块） | 浏览器（WebAssembly） |
| 打开数据库 | `new Database('seed.db')` | `const SQL = await initSqlJs(); new SQL.Database(arrayBuffer)` |
| 查询 API | `db.prepare(sql).all(...)` | `db.prepare(sql); stmt.bind([...]); while(stmt.step()){}` |
| **SQL 语法** | 标准 SQLite | **完全相同** |
| 持久化 | 直接读写文件 | IndexedDB 存储 ArrayBuffer |

**关键结论**：现有所有 SQL 查询语句（`kline_daily` / `stock_list` 两张表）一行都不用改——sql.js 是同一套 SQLite 引擎编译成 WebAssembly。差异仅在"打开数据库"那几行。

## 6. 数据策略：四层加载 + 零重复抽签

### 6.1 性能与体积实测（来自 `data/blind-seed.db`）

**查询性能**（随机抽 1 只取 120 根 K 线，3 次平均）：

| 股票池规模 | 100 只 | 500 只 | 1000 只 | 2000 只 | 4772 只 |
| --- | --- | --- | --- | --- | --- |
| 耗时 | 17ms | 14ms | 14ms | 14ms | 15ms |

**结论**：股票数量对查询性能几乎无影响。查询逻辑是"先随机抽 1 只 → 再取这只的 K 线"，股票总数只影响抽签（微秒级），取 K 线有索引恒定快。

**文件体积**（SQLite + VACUUM / gzip 传输）：

| 规模 | K 线数 | 本地存储 | 首次下载（gzip） | 加载体验 |
| --- | --- | --- | --- | --- |
| 100 只 | 11.8 万 | 14MB | 5MB | 秒开 |
| **500 只 ⭐基础池** | 57 万 | 66MB | **23MB** | **~10 秒** |
| 1500 只 | 172 万 | 198MB | 70MB | 30-60 秒，需进度条 |
| 3000 只 | 350 万 | 405MB | 142MB | 2-5 分钟，iOS 有限制 |
| 全部 4896 只 | 564 万 | 654MB | 230MB | 不可行（一次下载） |

> 注：上表"全部"为未排除低价股的原始过滤数；按 §6.6 完整筛选（再排除低价 <3 元 128 只）后为 **4772 只**，体积约 640MB，作为 2.6 年累积的上限。

### 6.2 重复概率分析（生日问题变体）

固定 500 只池不刷新时，训练 N 局后至少遇到 1 次重复的概率：

| 训练局数 | 10 | 30 | 50 | 100 |
| --- | --- | --- | --- | --- |
| 重复概率 | 8.7% | 58.8% | **92.1%** | 100% |

按每天 5 局算，~10 天到 50 局，体验明显下降。**用户顾虑成立**。

### 6.3 选定方案：方案 C · 大池 + 排除已用 + 渐进加载

核心机制：抽签时 `WHERE code NOT IN (SELECT code FROM trained_stocks)`，**数学上保证零重复直到用完一轮**。

容量计算：筛选后 4772 只（见 §6.6）÷ 每天 5 局 = **954 天 ≈ 2.6 年**才用完一轮，期间零重复。

### 6.4 四层加载架构

| 层 | 内容 | 体积 | 触发方式 |
| --- | --- | --- | --- |
| 第 1 层 · 内置包 | 100 只最活跃股 | 5MB | 随 App 打包，首次启动秒开 |
| 第 2 层 · 首启下载 | 500 只精选 | 23MB（gzip） | 启动后台拉取，~10 秒，完成后提示"数据就绪" |
| 第 3 层 · 自动补充 | 下一批 500 只 | 每次 +23MB | 本地用到 80% 时后台静默下载，用户无感 |
| 第 4 层 · 用户自主扩展 | 行业包 / 全量包 / 自选导入 | 按需 | 数据管理页主动选择 |

累积路径：500 → 1000 → 1500 → … → 4772（约 2.6 年），中途任意时刻可"重置已用记录"重新循环。

### 6.5 第 4 层 · 用户自主扩展（数据管理页）

三个入口：

1. **📈 行业包**：科技/消费/医药等，每包 50 只约 3MB。想练特定板块时下载。
2. **📦 全量包**：一次下载剩余所有股票，~150MB，WiFi 推荐，可后台下载不阻塞训练。
3. **✏️ 自选导入**：用户输入股票代码（如 600519）按需加载单只，给重度用户留口子。

页面实时显示：`1,250 / 4,896 只 · 已训练 68 只 · 还可练 1,182 局 · 本地占用 158MB`。

### 6.6 股票筛选规则

从 5148 只中排除（`sqlite3` 实测）：

| 排除项 | 数量 | 规则 |
| --- | --- | --- |
| ST 股 | 178 | `name LIKE '%ST%'` |
| 低价股 | 128 | 最新收盘价 < ¥3 |
| 银行/券商/保险/红利 | 70 | `name LIKE '%银行%' OR '%证券%' OR '%保险%' OR '%红利%'` |
| 新股 | 待定 | `list_date` 字段空，改用 K 线最早日期 > 2023-01-01 识别 |
| **剩余可保留** | **4772** | — |

500 只精选的筛选优先级：日均成交额 TOP（流动性好=短线活跃）→ 行业分散 → 价格区间 3-100 元为主。

### 6.7 核心算法（零重复抽签 SQL）

```sql
-- 本地表：记录已训练过的股票
CREATE TABLE trained_stocks (
  code TEXT PRIMARY KEY,
  trained_at INTEGER
);

-- 抽签：从本地池排除已训练的
SELECT code FROM stock_list
WHERE code IN (
  SELECT DISTINCT code FROM kline_daily  -- 仅本地已有的股票
)
AND code NOT IN (SELECT code FROM trained_stocks)
ORDER BY RANDOM() LIMIT 1;

-- 训练结束后：标记已用
INSERT INTO trained_stocks (code, trained_at)
VALUES (?, strftime('%s','now'));
```

**关键约束**：所有扩展下载的新股票自动进入抽签池，已用排除机制对所有数据一视同仁——不管内置的、自动补的、还是用户主动下的，用过一次就不再出现，直到用户手动"重置已用记录"。

## 9. 风险与权衡

### 9.1 需砍掉的功能（符合约束）

- **"数据管理"页的自动同步功能**：去掉（符合 AGENTS.md §2.2"盲训不主动更新"约束）。保留为"用户主动扩展"入口（第 4 层）。
- **用户自导股票池的文件路径配置**：移除（手机无文件系统概念），改为应用内输入 code。
- **`STOCK_TRADING_DB_PATH` 等环境变量覆盖**：移除，改为内置包 + 应用内下载。

### 9.2 技术风险

| 风险 | 影响 | 缓解 |
| --- | --- | --- |
| iOS Safari IndexedDB 配额 | 64GB 设备约 4-6GB，本方案最大累积约 640MB，安全 | 监控用量，超 80% 提示清理 |
| sql.js 首次加载 WASM | ~1MB，首次访问略慢 | service worker 缓存，二次秒开 |
| 大包下载（150MB+）影响体验 | 全量包下载耗时 | 限 WiFi + 后台下载 + 进度条 |
| iOS PWA 后台下载不稳定 | 自动补充可能中断 | 下次启动时检查并续传 |

### 9.3 待后续实现计划确认的细节

- 行业包的分类标准（申万一级？自定义？）。
- 自选导入的股票是否也要走筛选规则（排除 ST/低价）。
- 下载大包时是否强制限制仅 WiFi。
- "重置已用记录"是否需要二次确认弹窗。

## 10. 不做的事（YAGNI）

- ❌ 不做后端服务器（坚持全本地，符合 AGENTS.md §3）。
- ❌ 不做云同步 / 账号系统（一切本地）。
- ❌ 不做推送通知（PWA 限制 + 训练工具不需要）。
- ❌ 不做交易建议（AGENTS.md §3，本项目是训练工具）。
- ❌ 不主动同步行情数据（"盲"训核心约束）。
- ❌ 不引入与其他项目的 npm 依赖 / 共享代码包（AGENTS.md §2.4 / §3）。

## 11. 未来升级路径

PWA → React Native + Expo（若需正式 App Store 上架）：

- 业务逻辑层（tradingEngine / sampleFactory / types / store）零改动。
- UI 组件层重写为 RN 组件。
- sql.js 换成 `expo-sqlite`（同样是 SQLite，SQL 语句复用）。
- 数据分层策略完全沿用。

这是选择 PWA 的战略价值：**低沉没成本验证产品，未来可平滑升级**。
