# 贪心上涨段标签与经典策略 Benchmark 方案

**版本：** v1.1  
**状态：** 部分已实施  
**日期：** 2026-05-10  
**目标模块：** 模型训练 / 标签生成 / 策略回测

## 1. 背景问题

当前“策略打标”把经典指标策略和 ML 标签生成混在一起，导致三个问题：

1. 经典策略本身已经是交易规则，直接用于回测更合理，不应该先生成标签再训练模型。
2. 现有指标打标没有覆盖真正有效的上涨波段，标签质量差，模型学习目标不清晰。
3. 过多参数让打标配置变复杂，且参数之间容易逻辑冲突。

新的方向是拆分两件事：

1. **ML 标签生成**：用贪心算法找历史中可交易、空间足够、持续性足够的上涨段。
2. **经典策略 Benchmark**：MA/MACD/BOLL/突破等策略直接做回测和模型对照，不进入 ML 标签生成链路。

> 2026-05-10 状态说明：当前代码里的 `greedy_uptrend_segment_v1` 已经演进为“上涨趋势波段配对”实现，不再是纯从左到右的简单贪心。实现包含趋势段构造、Swing Low/High、周线锚、L1 trend filter、候选评分和 beam search。本文档保留“贪心上涨段”作为历史命名和业务入口背景，后续产品文案建议改为“上涨趋势波段”。

## 2. 核心原则

1. 标签不是为了复刻某个指标，而是为了标注“值得模型学习识别的上涨段”。
2. 交易是时间序列决策，采用从左到右的贪心扫描，不做全局回退优化。
3. 如果候选买点之后找不到合适卖点，或后续涨幅不达标，则直接放弃该买点。
4. 经典指标策略只作为可解释交易策略与 benchmark，不再作为 ML 标签配置项。
5. 所有标签必须满足可交易约束，避开一字板、连板、成交不足等无法执行场景。

## 3. 模块一：贪心上涨段标签生成器

### 3.1 目标

从历史行情中找到一组非重叠、顺序发生的 `buy-sell` 交易对，用作 ML 监督标签。

标签代表的是：

1. `buy`：上涨段中可交易低点或启动点。
2. `sell`：该段上涨空间兑现后的可交易退出点。
3. `pair`：一段完整、可执行、有训练价值的上涨波段。

### 3.2 硬性入选条件

候选上涨段必须同时满足：

1. `MA20` 向上：至少连续 3 个交易日斜率为正。
2. 持续时间：买点到高点至少 5 个交易日。
3. 上涨空间：买点到后续高点涨幅至少 8%。
4. 可交易：买点不能是一字板、连续涨停、成交额过低或明显无法买入。
5. 可退出：卖点不能是跌停无法卖出，也不能缺失成交量。

若任何条件不满足，放弃该候选买点。

### 3.3 贪心扫描流程

1. 按交易日从左到右扫描单只股票。
2. 当 `MA20` 开始持续向上时，进入候选趋势段。
3. 在候选趋势段内寻找可交易买点：
   - 价格靠近 `MA5/MA10/MA20` 支撑；
   - 或趋势启动后第一次有效回踩；
   - 或放量突破后回踩不破。
4. 买点确认后继续向右扫描未来行情。
5. 如果未来高点涨幅 `< 8%`，放弃该买点。
6. 如果未来持续时间 `< 5` 个交易日，放弃该买点。
7. 如果达到空间与持续要求，继续寻找卖点。
8. 找到合适卖点后落一组 `buy-sell` 标签。
9. 从卖点之后继续扫描下一段，交易对不重叠。

### 3.4 卖点规则

卖点不是“最高点预测”，而是上涨段兑现后的可执行退出点。优先级如下：

1. 达到 8% 以上涨幅后，跌破 `MA5`。
2. 达到 8% 以上涨幅后，`MA5` 拐头向下。
3. 高点后回撤超过阶段涨幅的 30%-40%。
4. 出现放量长上影或高位滞涨。
5. 达到最大持仓天数仍未触发退出，则选择最后一个可交易日退出。

如果买点之后没有满足卖点规则的可交易退出点，则放弃该买点，不生成半截标签。

### 3.5 不可交易过滤

买点过滤：

1. 当日一字涨停。
2. 买点前连续涨停达到阈值（默认 2 天）。
3. 开盘涨停且全天封板。
4. 日均成交额低于阈值。
5. 新股上市历史不足最小 bars。

卖点过滤：

1. 当日跌停且无法成交。
2. 成交量异常缺失。
3. 停牌或价格数据异常。

### 3.6 输出数据

每组标签输出：

1. `pair_id`
2. `pair_seq`
3. `buy_date`
4. `sell_date`
5. `buy_price`
6. `sell_price`
7. `return_pct`
8. `holding_days`
9. `max_drawdown_pct`
10. `exit_reason`
11. `tradability_flags`
12. `run_meta`

推荐策略名：

`greedy_uptrend_segment_v1`

## 4. 模块二：经典策略 Benchmark

### 4.1 定位

经典指标策略直接用于回测和 benchmark，不用于生成 ML 标签。

它们回答的问题是：

1. 不训练模型，单靠规则能赚多少？
2. ML 模型是否真的超过经典策略？
3. 哪些市场环境下经典策略更稳？

### 4.2 第一批内置策略

1. **MA Cross**
   - 买入：`MA5 > MA20`
   - 卖出：`MA5 < MA20`

2. **MA20 Trend Pullback**
   - 买入：`MA20` 向上，价格回踩 `MA5/MA10` 附近
   - 卖出：跌破 `MA5` 或 `MA10`

3. **MACD Above Zero**
   - 买入：MACD 水上金叉
   - 卖出：MACD 死叉

4. **Breakout**
   - 买入：突破近 N 日高点且成交量放大
   - 卖出：跌破突破位或跌破短均线

5. **BOLL Trend**
   - 买入：中轨向上，价格回踩中轨后重新上行
   - 卖出：跌破中轨或上轨冲高回落

### 4.3 Benchmark 输出

每次回测输出统一评估卡：

1. 样本区间
2. 交易次数
3. 胜率
4. 盈亏比
5. 平均持仓天数
6. 最大回撤
7. 总收益 / 年化收益
8. 不可交易过滤次数
9. 失效场景摘要

## 5. 页面调整

### 5.1 废弃当前“指标策略打标配置”

现有策略打标页面不再承担 MA/MACD/BOLL 等指标策略配置。

它应改为：

1. 上涨段标签生成入口。
2. 小批量验证。
3. 标签质量审查。
4. 标签覆盖率报告。

### 5.2 新增“经典策略回测 / Benchmark”模块

新模块能力：

1. 选择单个经典策略。
2. 选择多策略组合。
3. 设置回测区间、标的范围、交易约束。
4. 输出 benchmark 排名。
5. 与 ML 模型评估报告对比。

## 6. 数据边界

### 6.1 ML 标签表

`signal_candidates` 只接收：

1. `greedy_uptrend_segment_v1`
2. 人工审核后的标签
3. 明确用于 ML 的标签版本

### 6.2 策略回测表

经典策略回测写入独立结果表：

1. `strategy_definitions`
2. `strategy_backtest_runs`
3. `strategy_backtest_trades`
4. `strategy_benchmark_reports`

经典策略交易结果不写入 ML 标签表。

## 7. 验收标准

### 7.1 贪心上涨段标签生成器

1. `buy-sell` 完整率 >= 99%。
2. 所有标签满足 `MA20` 向上、持续 >= 5 天、空间 >= 8%。
3. 找不到合适卖点或涨幅不达标时，不生成任何标签。
4. 可交易过滤有明确统计：连板、一字板、成交不足、跌停不可卖。
5. 抽样审查中，一只股票能展示多段完整交易对。

### 7.2 经典策略 Benchmark

1. 每个策略能独立回测。
2. 多策略组合能输出合并结果。
3. 回测结果不污染 ML 标签数据。
4. ML 模型报告能引用 benchmark 结果做对照。

## 8. 实施顺序

1. 删除当前策略打标中的经典指标配置入口。
2. 实现 `greedy_uptrend_segment_v1` 标签生成器。
3. 调整标签审查页为上涨段交易对审查。
4. 新增经典策略库与回测 CLI。
5. 新增 Benchmark 页面与结果表。
6. 模型评估报告接入 benchmark 对比。

## 9. 当前结论

新的分工应固定为：

1. **贪心上涨段标签生成器**：负责给 ML 提供高质量上涨段标签。
2. **经典策略库**：负责直接交易回测和 benchmark。
3. **ML 模型**：负责学习上涨段出现前的因子共性，并证明自己能超过 benchmark。

这能避免“用策略打标再训练同类策略”的循环问题，也能让模型训练目标更接近真正想捕捉的市场机会。

## 11. 与反转 / 反弹打标的关系（2026-05-10）

本方案只负责顺势上涨段标签，核心假设是买点已经处于或接近上涨趋势中。

大级别反转 / 大级别反弹属于另一类监督目标：

1. 前置状态是下跌、低位震荡或弱势修复，而不是已确认上涨趋势。
2. 买点更依赖结构低点、止跌确认、未来路径验收，而不是 MA20 持续上行。
3. 样本分布更稀疏，应该独立 `source_strategy`、独立数据集、独立模型先验证。
