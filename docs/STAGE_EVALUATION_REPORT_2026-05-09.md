**项目：** Stock Trading Simulator / 人机协同交易系统  
**评估日期：** 2026-05-09  
**评估范围：** 系统功能、架构、算法效率、文档记录质量  
**依据：** 当前仓库代码、有效文档、核心实现抽查、构建与测试结果

## 1. 总体结论

项目已经从“股票交易模拟器”演进为一个本地桌面化的人机协同量化研究平台。当前系统不只是 K 线训练工具，而是形成了以下闭环：

```text
行情数据 -> 候选信号/标签 -> 数据集版本 -> 特征构建 -> 模型训练
       -> 回测/Benchmark/Walk-Forward -> 模型部署 -> 信号反馈/再训练

真实历史 K 线 -> 盲训会话 -> 行为记录 -> 会话复盘 -> 训练总览
```

综合判断：

1. **功能完整性：较高。** 数据管理、盲训、训练复盘、模型训练、策略验证、模型部署、Alpha 研究和 AI 助手已经进入主导航，产品形态明显超过普通模拟盘。
2. **架构专业性：中上。** Electron + React + SQLite + Python CLI 的单机架构符合个人桌面工具定位；IPC 分域、模型域拆分和 `PlatformResult<T>` 方向正确，但契约统一尚未完成。
3. **算法专业性：中上。** 已具备多版本特征、LightGBM/CatBoost、Optuna、时间序列切分、Walk-Forward、经典策略 Benchmark、回测成本与 A 股交易约束；但组合构建、风险归因、因子稳定性生产监控仍不足。
4. **算法效率：可用于个人研究，未达到生产级高频/批量服务。** 离线训练和批量回测适合当前定位；CatBoost 信号扫描通过 Python CLI 批量回退，功能可用但不是实时服务化最优路径。
5. **文档记录质量：内容丰富但索引漂移明显。** 文档数量和深度充足，但部分 README/索引仍引用旧文件名或旧模块数，和代码实况不同步，需要一次文档治理。

阶段评分：

|---|---:|---|
| 功能覆盖度 | 8.0 / 10 | 核心业务闭环成型，模型部署和策略验证已进入可用阶段 |

| 算法专业性 | 7.5 / 10 | 研究工具链较完整，但风险/组合/监控仍薄弱 |
| 算法效率 | 6.5 / 10 | 离线任务可接受，实时/批量推理仍有瓶颈 |
| 工程质量 | 6.5 / 10 | 构建和 Python 测试通过，Lint 与契约校验失败 |

## 3. 架构评估

### 3.1 当前架构形态

系统采用本地桌面单机架构：

```text
React Renderer
  -> Preload Bridge / window.electronAPI
  -> Electron Main Process
  -> SQLite / Local File System
  -> Python CLI Subprocess
```

主要技术栈：

| 层级 | 技术 |
|---|---|
| 前端 | React 19 + TypeScript + Vite |
| 桌面壳 | Electron 41 |
| 持久化 | better-sqlite3 + 本地文件系统 |
| 图表 | klinecharts |
| 算法 | Python、pandas、numpy、LightGBM、Optuna、scikit-learn、CatBoost 代码路径 |
| 任务运行 | Electron 主进程 spawn Python CLI |

### 3.2 架构优点

1. **产品定位与架构匹配。** 项目明确不做自动下单、不做高频交易，因此本地单机桌面架构是合理选择，部署成本低，用户数据可本地保存。
2. **分层边界基本清楚。** Renderer 负责交互，Preload 负责桥接，Main 负责 IPC/SQLite/Python 调度，Python 负责训练、特征、回测、研究计算。
3. **模型域正在拆分。** 已有 `modelDatasetIpc.ts`、`modelCliRunner.ts`、`modelFeatureCalculator.ts`、`modelSignalInferenceService.ts`、`modelFeedbackRetrainingService.ts` 等拆分。
4. **持久化模型较完整。** SQLite 表覆盖训练会话、候选信号、数据集、特征任务、模型任务、模型版本、评估、信号事件、反馈、再训练等关键对象。
5. **可追溯性意识强。** 模型产物、评估报告、manifest、任务 stdout/stderr、回测报告、推荐记录都已落库或落文件。

### 3.3 架构问题

1. **IPC 契约尚未统一完成。** 文档宣称主链路正在统一到 `PlatformResult<T>`，但契约校验脚本当前失败，preload/global 中大量接口仍是 `UnknownRecord` 或旧返回形态。
2. **模型域组装层仍偏重。** `src/main/ipc/model.ts` 已拆出子模块，但仍包含不少策略统计、归因、数据集策略建议、盲训收益关联等业务逻辑，后续可继续下沉到 service。
3. **TS/Python 特征逻辑双实现。** `modelFeatureCalculator.ts` 为实时推理维护了一份 TypeScript 特征计算逻辑，Python `FeatureBuilder` 是更完整权威实现。双实现会带来特征漂移风险。
4. **Python CLI 适合离线任务，不适合高频实时推理。** 当前 spawn CLI 的任务编排简单可靠，但 CatBoost 批量扫描回退到 Python CLI 后，实时性受进程启动和逐批计算影响。
5. **数据库 schema 与文档之间存在漂移。** 当前代码新增 `model_recommendations`、指数数据能力等，但部分主文档仍未完全反映。

### 3.4 架构专业性结论

当前架构适合“个人研究 + 桌面交易训练 + 辅助信号提醒”阶段。它不是机构级量化生产架构，也不应在当前阶段强行服务化。下一阶段最重要的不是引入复杂后端，而是完成：

1. IPC 契约统一。
2. Python 特征权威化。
3. 模型域 service 边界继续收敛。
4. 离线任务、实时推理、回测验证三条运行路径的口径对齐。

## 4. 算法与效率评估

### 4.1 已具备的算法能力

#### 标签与候选

1. 贪心上涨段标签 `greedy_uptrend_segment_v1`。
2. Swing low/high 波段配对标注。
3. 趋势过滤：MA、ADX、成交额、上市天数、涨停连板等。
4. 不可交易过滤和人工审核链路。
5. 数据集冲突策略：保留全部 / 同 bar 最高分。

#### 特征工程

1. v001-v011 特征规格，覆盖价格、趋势、动量、波动、量价、截面、行业、隔夜、尾盘 15m 代理特征。
2. SQLite dataset_items 到 parquet split 的特征构建链路。
3. 严格真实数据模式 `strict_real_dataset`，可避免 mock fallback 混入生产训练。
4. 时间序列切分：70% train、10% valid、20% test。

#### 模型训练

1. LightGBM Trainer。
2. CatBoost Trainer。
3. Optuna 超参搜索。
4. 时间序列 CV / day-blocked CV。
5. 特征重要性与训练报告输出。
6. 模型 artifact 和 native model 双产物。

#### 策略验证

1. BacktestEngine 支持 D+1 open 入场。
2. 按 D+N close 保守退出。
3. D+N high 乐观上界仅作参考。
4. 交易成本包含佣金、印花税、过户费、滑点。
5. A 股涨停买不进、缺后续行情跳过等执行约束。
6. 阈值敏感度分析与阈值优化。
7. MA、MACD、BOLL、Breakout 经典策略 Benchmark。
8. Walk-Forward 滚动窗口训练和 OOS 评估。

#### Alpha 研究

1. Pearson IC、RankIC。
2. 月度 RankIC IR。
3. 分箱收益与单调性。
4. 相关性矩阵与高相关因子对。
5. IC 时间序列、分布、衰减。
6. train/valid/test split IC 对比。

### 4.2 算法专业性优点

1. **防未来函数意识较强。** 特征审计器检查 `shift(-N)`、`rolling(center=True)`、`bfill` 等风险；Python 测试也覆盖了 leakage audit。
2. **回测口径相对严谨。** 回测说明明确区分候选信号、可成交交易、乐观上界，不把最高价止盈空间当作主收益。
3. **A 股约束进入回测。** 涨跌停、T+1、交易成本等约束比普通回测更接近真实交易。
4. **引入了 Walk-Forward。** 这比单次 train/test split 更专业，是判断模型是否稳定的必要基础。
5. **有经典策略对照。** 不再只看模型 AUC，而是能和可解释规则策略进行收益对比。
6. **标签质量被重视。** 从人工审核、标签审计、数据集冻结到反馈回灌，已经形成数据闭环雏形。

### 4.3 算法专业性风险

1. **模型高指标仍需警惕过拟合。** `docs/CHANGELOG_20260509.md` 记录了高胜率、宽松打标、CatBoost 强拟合等现象。当前可判断“未发现明显未来函数”，但不能等同于“模型已具备稳定可交易 alpha”。
2. **Benchmark 公平性仍需优化。** 当前 Benchmark 的 ML 对照路径标记 `disable_threshold` 和 `disable_topn`，会使 ML 信号暴露与经典策略不完全等价。后续应增加固定交易频率、固定资金约束、统一风险预算下的对照。
3. **组合与仓位仍弱。** 回测主指标仍偏信号质量评估，尚未完整模拟跨日持仓重叠、资金占用、仓位滚动、组合暴露。
4. **风险管理不足。** 目前有最大回撤和成本，但缺 VaR/CVaR、行业暴露、市值暴露、Kelly/风险预算、熔断规则。
5. **因子稳定性离生产监控还有距离。** Alpha 研究中已有 IC 时间序列和衰减计算，但还未形成定期监控、告警、因子下线机制。
6. **特征口径可能漂移。** Python feature builder 支持 v011 和截面/行业/日内特征，TypeScript 实时推理只覆盖部分实时特征；文档也记录 v005+ 可能缺少横截面实时特征。
7. **mock fallback 有误用风险。** FeatureBuilder 在非 strict 模式下会回退 mock 数据。研发调试方便，但生产训练应默认开启 strict real dataset 或在 UI 明确警示。

### 4.4 算法效率评估

| 模块 | 当前复杂度/效率 | 评价 |
|---|---|---|
| 特征构建 | pandas 分组计算，离线批处理 | 适合个人研究；全市场大样本时需关注内存和预加载开销 |
| v011 特征 | 多 rolling + swing 遍历 | 可接受，但部分 swing 距离逐行循环，后续可向量化 |
| LightGBM 训练 | Optuna + CV + final model | 训练质量较好，时间成本取决于 trials 和样本量 |
| CatBoost 训练 | CPU + Optuna | 专业但较重，需默认正则和 trials 上限 |
| 回测 | pandas + SQLite kline lookup | 对测试集和中等规模信号可用；大规模反复 threshold sweep 会重复计算 |
| Walk-Forward | 每窗口重训 + OOS 回测 | 专业但重，应作为离线验证任务 |
| 实时 LightGBM 推理 | TS tree_dump 本地推理 | 响应较好，但与 Python 特征口径需校准 |
| 实时 CatBoost 推理 | Python CLI batch，每批 40 只 | 功能可用，效率一般；建议常驻 worker |

效率结论：

当前系统适合“每日/盘后批量扫描 + 手工复核 + 离线训练验证”，不适合“高频、低延迟、全市场秒级重算”。这与产品非目标一致，但下一阶段若要增强准实时提醒，应优先做 Python 常驻 worker 或统一推理服务，而不是继续扩大按次 CLI 调用。

1. **文档数量充足。** 当前非 archive 文档约 6000 行，涵盖 BRD、PRD、架构、路线图、中期报告、策略验证、模型训练、算法方案、用户手册等。
2. **阶段记录意识强。** `MIDTERM_REPORT.md`、`ROADMAP.md`、`RECENT_CHANGES_2026-05-04.md`、`CHANGELOG_20260509.md` 都提供了阶段上下文。

4. **算法文档有专业深度。** 趋势分割、贪心上涨段、模型训练、策略验证文档都体现了标签、特征、回测和评估口径意识。

### 6.2 文档问题

1. **有效文档索引漂移。** `docs/README.md` 引用了 `ML_MODEL_TRAINING.md`、`QUANT_OPTIMIZATION_RUNBOOK.md`、`FEATURE_EXPLORATION_FACTOR_MINING_TECH_SPEC.md` 等无当前同名文件或已带 `[已完成]`、`[已归档]`、`[已废弃]` 前缀的文件。

3. **文档状态标记不统一。** 同一类文档同时存在“当前有效”“已完成”“已归档”“已废弃”多套命名方式，降低检索可信度。
4. **计划与实况混杂。** 部分文档同时写“缺失”和“已完成”，靠后文追加状态更新修正，阅读成本较高。
5. **新增能力未及时吸收进架构真相文档。** `model_recommendations`、指数数据库、推荐复盘筛选、CatBoost 扫描回退等 2026-05-09 变更需要并入主架构和数据流文档。

### 6.3 文档质量结论

文档不是缺少内容，而是缺少“单一可信入口”。当前更像快速迭代中的研究日志 + 方案文档集合。下一阶段应做一次轻量治理：

1. 更新 `docs/README.md` 为唯一文档导航。
2. 把已归档、已废弃、已完成文档移入 archive 或从“当前有效”清单移除。
3. 更新根 README 与 PRD 中的前台模块清单。
4. 将 2026-05-09 变更吸收到 `ARCHITECTURE.md` 和 `ROADMAP.md`。
5. 给每篇当前有效文档增加：状态、最后更新时间、对应代码入口、是否代表当前实现。

## 9. 专业性最终判断

当前项目已经具备“个人量化研究平台 + 交易训练平台”的专业雏形：

1. **不是玩具模拟器。** 真实行情、A 股交易约束、模型训练、回测、Walk-Forward、反馈再训练都已存在。
2. **不是生产级量化交易系统。** 缺少组合构建、风险归因、执行算法、实时服务、监控告警和严格实验平台。
3. **最有价值的差异化方向是人机协同。** 盲训提升人的决策质量，模型提升机会发现效率，反馈闭环让两者互相增强。
4. **下一阶段成败关键不在继续堆功能，而在校准口径。** 训练、回测、实时信号、文档和类型契约必须对齐，否则功能越多，解释成本越高。

建议阶段目标从“功能扩张”切换为：

```text
可验证、可复现、可解释、可维护
```

当工程门禁、文档索引、回测公平性和实时特征口径都统一后，系统才能从 Alpha 研究平台稳定迈向可长期使用的个人交易辅助平台。
