# 股票交易模拟应用 - 业务需求分析与技术选型报告

## 一、项目概述

### 1.1 项目背景
本项目旨在开发一款股票交易模拟应用，帮助用户通过模拟交易来提升个人交易技能。应用的核心价值在于提供一个无风险的练习环境，让用户能够在历史数据上进行交易决策练习，并通过系统化的分析来改进交易策略。

### 1.2 目标用户
- 个人投资者：希望提升交易技能的散户投资者
- 交易学习者：正在学习技术分析和交易策略的初学者
- 策略验证者：希望验证自己交易想法的量化爱好者

### 1.3 核心价值主张
- **消除认知偏差**：通过匿名化股票信息，消除用户对特定股票的先入为主判断
- **即时反馈**：每笔交易决策都能得到即时的绩效反馈
- **策略对比**：与基准策略和推荐策略进行对比，发现改进空间
- **模式识别**：识别成功和失败交易的共同模式，帮助用户改进

## 二、业务需求分析

### 2.1 核心模拟功能

#### 2.1.1 K线交易决策
| 功能点 | 描述 | 优先级 |
|--------|------|--------|
| K线逐根交易 | 用户可以在每根K线上做出交易决策（买入/卖出/持有） | P0 |
| 收盘价执行 | 所有交易在K线收盘价执行 | P0 |
| 历史数据遮蔽 | 隐藏当前K线之后的价格走势 | P0 |
| 股票匿名化 | 移除股票代码和公司名称，消除先验知识偏差 | P0 |
| 随机选股 | 从预设股票池中随机选择股票和起始点 | P0 |
| 跳过功能 | 允许用户跳过当前股票，作为基准对比 | P1 |

#### 2.1.2 交易记录与标记
| 功能点 | 描述 | 优先级 |
|--------|------|--------|
| 决策标记 | 标记每根K线的交易决策（买/卖/持有） | P0 |
| 入场/出场记录 | 记录完整的交易周期 | P0 |
| 持仓管理 | 跟踪当前持仓状态和盈亏 | P0 |
| 交易日志 | 记录用户的交易理由和备注 | P1 |

#### 2.1.3 绩效计算
| 指标 | 描述 | 优先级 |
|------|------|--------|
| 盈亏计算 | 计算每笔交易和总体盈亏 | P0 |
| 胜率统计 | 计算盈利交易占比 | P0 |
| 风险收益比 | 计算平均盈利/平均亏损比率 | P0 |
| 最大回撤 | 计算最大资金回撤幅度 | P1 |
| 夏普比率 | 计算风险调整后收益 | P2 |

### 2.2 模拟约束条件

#### 2.2.1 数据约束
```
约束规则：
1. 时间遮蔽：用户只能看到当前K线及之前的数据
2. 身份匿名：股票代码替换为随机ID，公司名称隐藏
3. 随机起始：每次练习从随机时间点开始
4. 固定周期：预设K线周期（日线/周线/60分钟线等）
```

#### 2.2.2 交易约束
```
交易规则：
1. 执行价格：收盘价
2. 交易方向：做多（暂不支持做空）
3. 仓位管理：固定金额或固定比例
4. 交易成本：可配置手续费和滑点
```

## 四、技术栈选型建议

### 4.1 方案对比

#### 方案A：SwiftUI 原生应用
```
技术栈：
- 前端：SwiftUI + Swift
- 后端：Swift / Python Core
- 数据库：Core Data / SQLite
- 可视化：Swift Charts / 自定义渲染

优点：
✅ 最佳的macOS原生体验
✅ 性能极致
✅ 内存占用低
✅ 与系统集成度高

缺点：
❌ 仅限macOS，无法跨平台
❌ 金融库生态不如Python丰富
❌ 开发效率相对较低
```

#### 方案B：Electron 跨平台应用
```
技术栈：
- 前端：React/Vue + TypeScript
- 后端：Node.js + Python子进程
- 数据库：SQLite / IndexedDB
- 可视化：Lightweight Charts + ECharts

优点：
✅ 开发效率高
✅ 生态丰富
✅ 跨平台支持
✅ 热重载开发体验好

缺点：
❌ 应用体积大（100MB+）
❌ 内存占用高（300-500MB）
❌ 启动速度较慢
```

#### 方案C：Tauri 轻量跨平台应用（推荐）
```
技术栈：
- 前端：React/Vue + TypeScript
- 后端：Rust + Python子进程
- 数据库：SQLite（通过Rust）
- 可视化：Lightweight Charts + ECharts

优点：
✅ 应用体积小（10-20MB）
✅ 内存占用低（50-100MB）
✅ 启动速度快
✅ 安全性高（Rust内存安全）
✅ 前端技术栈灵活

缺点：
❌ 需要学习Rust（后端逻辑）
❌ 生态相对年轻
```

## 六、风险与缓解措施

### 6.1 技术风险
| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 数据源不稳定 | 数据获取失败 | 多数据源备份（AKShare + Tushare + Baostock） |
| Python性能瓶颈 | 响应延迟 | 关键路径用Rust实现、数据预加载 |
| Tauri生态不成熟 | 功能受限 | 关键功能用Python实现、保持架构灵活 |

### 6.2 业务风险
| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 用户学习成本高 | 用户流失 | 提供引导教程、简化操作流程 |
| 数据准确性问题 | 用户信任度下降 | 数据校验、异常检测 |

## 七、总结与建议

### 7.1 核心推荐
1. **数据获取**：AKShare（主） + Tushare（备）
2. **回测引擎**：VectorBT
3. **技术分析**：TA-Lib
4. **可视化**：Lightweight Charts + ECharts
5. **应用框架**：Tauri 2.0
6. **前端框架**：React + TypeScript
7. **数据处理**：Python + Pandas

### 7.2 开发建议
1. **先做MVP**：聚焦核心模拟功能，快速验证
2. **数据先行**：优先完成数据获取和存储模块
3. **模块解耦**：前端、后端、数据处理保持独立
4. **渐进增强**：功能逐步添加，保持稳定性

### 7.3 下一步行动
1. 搭建Tauri项目骨架
2. 集成AKShare数据获取
3. 实现基础K线展示
4. 开发交易决策记录功能
5. 实现绩效计算模块
