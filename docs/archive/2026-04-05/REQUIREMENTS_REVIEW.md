# 股票操盘训练系统 - 代码校验后的需求评审

**版本：** v0.2  
**日期：** 2026-04-05

## 一、评审结论

当前最大问题不是“需求不清楚”，而是 **文档把未来目标写成了当前能力**，导致代码、文档、预期三者不一致。

这次按代码校验后的结论是：

1. 当前产品的真实核心是 `盲训工作台 + 风格打标 + 本地数据同步`。
2. `训练记录`、`数据集与模型` 目前仍偏 preview 页面。
3. `复盘评分 / 扩样 / 数据集版本 / 模型训练` 仍属于后续路线，不应继续写成当前已在做的主线能力。

## 二、代码与文档不匹配的地方

### 2.1 盲训样本来源

文档原表述：

- 强调匿名样本池、市场状态抽样、真实样本训练

代码实际：

- 盲训样本主要由前端随机生成
- 市场状态 `uptrend/downtrend/range/volatile` 也是前端模拟
- 尚未形成真实样本池服务

结论：

- 当前应把“真实样本池盲训”定义为下一步 P0，而不是现状描述。

### 2.2 训练记录与复盘

文档原表述：

- 强调训练完成后的结构化复盘与行为评分

代码实际：

- session/action 会写入数据库
- 训练记录页仍是硬编码数组
- 复盘评分、行为标签、PnL 曲线未落地

结论：

- 当前只能说“已有 session 持久化基础”，不能说“已具备完整复盘系统”。

### 2.3 风格打标

文档原表述：

- 强调打标、扩样、候选审核、数据集构建

代码实际：

- 已有手工买点/卖点标注原型
- 标签可写入 `labels` 表
- 但样本仍主要是随机数据
- `no_action`、候选审核、数据集消费链路未打通

结论：

- 当前应把打标模块收敛为“真实 K 线上的手工标注能力”，先不把后续链路写成既成事实。

### 2.4 数据集与模型

文档原表述：

- 已进入数据集版本、模型版本、训练评估闭环

代码实际：

- 页面是 preview
- IPC 有 stub
- Python 特征构建仍使用模拟数据

结论：

- 当前只能定义为“保留入口和数据结构方向”，还不是产品主线能力。

## 三、这次对齐后，真正的核心需求点

### 3.1 核心需求 1：盲训必须从 demo 变成真实训练

需要对齐：

1. 使用本地真实 K 线数据生成样本。
2. 保持当前逐 bar、T+1、仓位和手续费规则。
3. 真实保存 session 和 action。
4. 匿名能力默认开启。

### 3.2 核心需求 2：打标必须从 demo 变成真实数据生产入口

需要对齐：

1. 打标页读取真实历史 K 线。
2. 标签读写数据库。
3. 页面展示已保存标签，而不是只展示内存状态。
4. 标签类型先收敛，不急着扩展复杂体系。

### 3.3 核心需求 3：历史记录必须成为真实回看入口

需要对齐：

1. 训练记录页直接读取真实 session 数据。
2. 至少能看到会话状态、交易数、收益结果、开始时间。
3. 不再保留默认 mock 数据。

### 3.4 核心需求 4：文档与工程边界必须统一

需要对齐：

1. 当前文档只描述当前可交付范围。
2. 后续能力显式放到路线图或后续范围。
3. 不再在 PRD 中混写“已做”和“准备做”。

## 四、需求优先级重排

### P0

1. 修复 build 和类型问题。
2. 盲训接真实样本。
3. 训练记录接真实数据。
4. 打标页接真实 K 线与真实标签。
5. 文档按当前实现边界重写。

### P1

1. 基础复盘指标。
2. `no_action` 标签。
3. 标签审核状态与简单筛选。
4. 数据集与模型页面改为真实元数据页。

### P2

1. 自动扩样。
2. 数据集版本化。
3. 模型训练和评估。
4. 统一 job 机制。

## 五、下一步建议

最合理的顺序不是继续加页面，而是先把主链路收紧：

1. `blind-training`
2. `training-history`
3. `style-label-workbench`
4. `desktop-shell-and-ipc`
5. `dataset-model-center`

也就是说，先把“训练”和“打标”变成真实系统，再去谈“建模闭环”。
